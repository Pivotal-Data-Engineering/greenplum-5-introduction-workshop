General Architecture

GPText 3.x

GPText Components

GPText consists of

  • Database functions in gptext schema
    • Some functions are in pl/pgsql for simplicity
    • Main interface functions are in C
  • Shared library gptext-3.x.x.so
    • Contains Postgres C functions
    • Uses libcurl.so to query Solr REST API
    • Uses libxml2.so to parse XML response from Solr REST API
  • SolrCloud
    • Zookeeper instances manage and monitor SolrCloud cluster
    • Solr and Lucene core jar libraries

Main Functions - Indexing

  1. Create index
    • Metadata operation in Solr, simple REST API call
  2. Index data
    • Function executes on segment level
    • Accepts stream of table tuples and passes them to the Solr instance
    • Solr performs analysis of text and adds to the index
  3. Commit index
    • Metadata operation in Solr, simple REST API call
  4. Drop index
    • Metadata operation in Solr, simple REST API call

Main Functions - Search

  • Search
    • REST API call performed by each segment
    • Each segment returns data separately (independent of other segments)
    • gptext.search ( <table-valued expression>, <index name>, <query>, <filter options>, <SOLR options> )
  • Highlighting of search terms
    • gptext.search ( <table-valued expression>, <index name>, <query>, <filter options>, 'hl=true&hl.fl=content' )

Syntax of a Search Query

Searches text index for <search query> and returns ID's and relevance scores of matching document 

SELECT * FROM gptext.search(TABLE( SELECT 1 SCATTER BY 1),

  idx_name,        /* Name of the index to search          */

  search_query,    /* Enhanced Solr/Lucene query
                      parser syntax                        */

  filter_queries,  /* Array to filter out search terms by
                      other columns in the index           */

  options          /* Solr options to limit data, order,
                      highlights, or return full results   */
);

Search Query - Lucene Syntax

Basic Lucene query syntax

Term match "test" OR "hello"
Field specification title:"The Right Way" AND text:"some text"
Wildcard searches te?t OR test*
Fuzzy searches (Levenshtein Distance) roam~ OR roam~0.8
Proximity searches "jakarta apache"~10
Range searches id:[5 TO 10] OR title:{Aida TO Carmen}
Term boosting jakarta^4 apache
Boolean searching AND, OR, NOT, +, -, etc
Grouping (jakarta OR apache) AND website

Search Query - Solr and UQP Syntax

Wild cards may be used in ranges field:[* TO 100]
Pure negative queries -inStock:false
Nested query parser support _query_:"{!dismax}how now brown cow"
Ordered Proximity searches 5n(dog cat) OR 5w(cat dog)
Nested Proximity searches 5n(dog AND 3n(cat mouse))
Wildcards in phrases "now now * brown cow"

  • hl and hl.fl : highlight search terms in the returned document
  • fl : returns extra fields, in addition to document id and score
  • defType : specify query parser to use for the search

Search Query - Joins

SELECT l.id,
       l.score,
       r."to"
FROM
       gptext.search(table(select 1 scatter by 1),
                     'gpuser.public.enron',
                     'content:2w(Phillips Petroleum)',
                     array['to:"Christine Stokes"',
                           'date:["2000-01-01T00:00:00Z" TO "2001-01-01T00:00:00Z"]'],
                     'hl=true&hl.fl=content&rows=1') l,
       enron r
WHERE
       l.id = r.id::text;
  • Allows you to return more than just the doc id and score
  • Query can have an arbitrary complexity - join with additional table or functions

Main Functions - Facet Search

Faceting breaks query results into multiple categories with a count of the number of documents in the index for each category.

  • Facet by: field, query, range
    • gptext.faceted_field_search() – search, faceted by fields
    • gptext.faceted_query_search() – search, faceted by queries
    • gptext.faceted_range_search() – search, faceted by defined ranges

Syntax : Faceted Search

SELECT * FROM gptext.faceted_field_search(
  idx_name,        /* Name of the index to search          */

  search_query,    /* Enhanced Solr/Lucene query
                      parser syntax                        */

  filter_queries,  /* Array to filter out search terms by
                      other columns in the index           */

  facet_fields     /* An array of field names to facet
                      - Use Greenplum  array notation      */

  facet_limit      /* Max number of results to be returned
                      for each aggregation (facet)         */

  min_results      /* Minimum number of results required
                      before an aggregation (facet) will be
                      returned (0 to return all facets)    */

  options          /* Solr options to limit data, order,
                      highlights, or return full results   */
);

GPText and SOLR differences

GPText vs SOLR

  • Distributed Solr - Highly scalable
    • Leverages GPDB MPP architecture to distribute nodes and shards across multiple hosts
  • SQL Interface
    • All Solr calls and functionality are exposed through SQL UDFs for a consistent interface through psql/pgadmin/etc
  • Ease of Configurability
    • GUC Configuration management
    • gptext-config
  • Integrated with GPDB and included in PDS (Pivotal Data Suite)!

GPText vs SOLR (cont.)

  • International/Social Media analyzer chains
    • Additional Analyzer Chains with custom Tokenizers/Filters to be able to accurately parse out International Text (including CJK characters) and Social Media (hashtags, emoticons, links, etc.)
  • Unified Query Parser
    • Combines Solr and Lucene query parsers into a single unified interface
  • Enables term vector support for text analytics with MADLib
    • LDA, K-Means Clustering, SVM

Two main analyzer chains

  • text_intl - international text support with WorldLexer improvements
  • text_sm - for use with social media and support for emoticons

Note: Search terms in queries are processed by the analyzer chain defined for an index

Analyzer Chain - text_intl

Getting ready for indexing

  • WLT - World Lexer Tokenizer
    • ​separates CJK characters from others and identifies symbols
  • CJKWF - Chinese/Japanese/Korean Word Filter
    • ​half/full width characters
  • LCF - Lower Case Filter
  • WLBF - Word Lexer Bigram Filter
    • generates Bigrams for any CJK characters
      • han, hiragana, katakana, hangul
  • SF - Stop word Filter
  • KMF - Keyword Marker Filter
    • marks words to prevent stemming (protwords.txt)
  • PSF - Porter Stemmer Filter

Analyzer Chain - text_sm

Getting ready for indexing

  • WT - Whitespace Tokenizer
  • SF - Stop word Filter
  • LCF - Lower Case Filter
  • ECTF - Emoticons Classifier Filter
    • uses the emoticons.txt file
  • TTF - Twitter Tokenizer
    • extends whitespace tokenizer
    • emoticons, hyperlinks, #hashtags, @users, etc

Configuring GPText

gptext-config

  • Allows editing of the following files
    • schema.xml
    • stopwords.txt
    • protwords.txt
    • synonyms.txt
    • elevate.xml
    • emoticons.txt
  • Distributes the specified configuration files to the specified directory (-d parameter)
  • Allows distribution of jar files (-j parameter)
  • Modifies jvm_options (-o parameter)
  • Appends a file to one of the config files (-a parameter)
  • Sets properties in the solr.xml config file (-k parameter)

Configuring GPText - options

gptext-config syntax 

  • gptext-config -f <file> -i <index> [-r] [-e] [-b]
  • gptext-config -d <path/file> -i <index>
  • gptext-config -j <path/jar>
  • gptext-config -o <jvm-options>
  • gptext-config -f <file> -i <index> -a <file to append>
  • gptext-config -k <solr key> -s <solr value>

Security

Users, Roles & Groups

About Roles and Privileges

  • Login Roles (formerly Users) and Groups
  • Not related to OS users and groups
  • Attributes to determine permission levels
  • Every system has a default superuser role (gpadmin)
  • Roles are given access privileges to databases, schemas, and objects
  • Roles can be members of other roles
  • Roles are defined at the system/server level, above databases

User Roles

  • User Account = role that can login
  • Default attributes for a new role:
    • NOSUPERUSER
    • NOCREATEDB
    • NOCREATEROLE
    • INHERIT
    • NOLOGIN (must explicitly grant LOGIN to user roles)

Syntax

  • CREATE ROLE : adds new role
  • ALTER ROLE : change privileges for an existing role
CREATE ROLE john
    WITH LOGIN
         RESOURCE QUEUE adhoc;

ALTER ROLE john
    WITH CREATEDB;

Group Roles

  • A role can be member of other roles
  • Members inherit object privileges of the parent role
  • Allows DBA to set object privileges in one place
  • Role attributes not inherited, but can SET ROLE to a parent role to access a role attribute
  • Group roles typically do not have LOGIN permissions
  • Use GRANT/REVOKE to grant/remove members from a role

Syntax

  • GRANT : add role to group
  • REVOKE : removes a role from a group
CREATE ROLE grp_admin
    WITH CREATEROLE CREATEDB;

GRANT grp_admin TO john, sally;

REVOKE admin FROM bob;

DB Object Privileges

About Object Privileges

  • Each object has an OWNER role having all privileges
  • All other roles must be granted privileges
  • PUBLIC privilege = object accessible to all roles
  • Use GRANT/REVOKE commands to manage privileges
  • Use DROP OWNED/REASSIGN OWNED for managing objects owned by disabled (LOGIN revoked) roles
GRANT ALL ON DATABASE mydatabase
   TO admin
   WITH GRANT OPTION;

GRANT SELECT ON TABLE mytable
   TO PUBLIC;
REVOKE INSERT UPDATE
   ON TABLE mytable FROM sally;

REASSIGN OWNED BY sally TO john;

DROP OWNED BY visitor;

Host-Based Authentication

pg_hba.conf

Host Based Authentication

  • Handles user authentication
  • The file is located in $MASTER_DATA_DIRECTORY
  • Comments are ignored
  • File is read line by line
    • First matching line is used and all subsequent lines are ignored
  • Pessimistic - if no grants, then deny access
  • To be able to access a Greenplum DB from an external host, the role/host pair has to be set in this file
    • If the pair is not declared, access will be denied, even if the role exists in Greenplum

  • Database name or list of comma separated names
  • all : tag specifying all databases
  • @<filename> : file containing list of database names, 1 per line
  • Role name or list of roles (separated by commas)
  • all: tag specifying all roles
  • +<role>: a group role where access is granted to all members
  • @<filename> : file containing list of roles, 1 per line

pg_hba.conf

Network Address

  • Does not apply for local connections
  • IPv4 and IPv6 supported
  • Two formats
    • CIDR-address
      • https://en.wikipedia.org/wiki/Classless_Inter-Domain_Routing
    • IP + Mask

pg_hba.conf

Authentication Method

trust No password asked, access is granted immediately
reject Access is denied/rejected
md5 Password required: hashed using md5 before sending to server
password Password required: transmitted in plain text
ldap Authentication using LDAP, encrypted or unencrypted
gss Authentication using GSSAPI and Kerberos
pam Uses the Linux Pluggable Authentication Method to authenticate the user (stacking of methods is possible)

pg_hba.conf

Examples

# allow the gpadmin user local access to all databases
# using ident authentication
local   all   gpadmin      ident               sameuser
host    all   gpadmin      127.0.0.1/32        ident
host    all   gpadmin      ::1/128             ident

# allow the 'dba' role access to any database from any host
# with IP address 192.168.x.x and use md5 encrypted passwords
# to authenticate the user
host    all   dba          192.168.0.0/16      md5

# allow the 'user1' role access to the databases db1 and db2 from any host
# and use no passwords to authenticate the user
host    db1,db2  user1     0.0.0.0/0           trust

# allow all the members of the 'gp-admin' group role access to the database vpdb from any
# host with IP address 192.168.x.x and use md5 encrypted  passwords to authenticate the user
host    vpdb    +gp-admin     192.168.0.0/16   md5

# allow all roles access to all databases from hosts with IP address 192.168.x.x and use
# ldap to authenticate the user
# NOTE: Greenplum role names must match the LDAP common name.
# NOTE: split across three lines for clarity. Must be on same line to take effect.
host    all   all   192.168.0.0/16 ldap
        ldapserver=usldap1 ldapport=1389
        ldapprefix="cn=" ldapsuffix=",ou=People,dc=company,dc=com"

pg_hba.conf

Applying Settings

  • Changes in the pg_hba.conf file will not take effect until you execute gpstop -u
    • Signals the database to read and apply settings in configuration files; does not stop the database
  • The order of entries in the file is very important. The first entry matching user's host/role pair wins 
host  all   user1   0.0.0.0/0	    trust
.
.
.
host  all   user1   192.168.0.0/16	md5  # This will not be applied
